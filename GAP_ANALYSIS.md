# QP-RS vs QP/C++ Gap Analysis

**Date**: 2026-06-15 (refreshed; originally 2026-06-10)  
**QP/C++ reference version**: v8.1.4 (April 2024)  
**Goal**: Make qp-rs a drop-in replacement / straightforward port target for QP/C++ applications

---

## Executive Summary

qp-rs covers the kernel layer (QF cooperative, QK preemptive, QXK extended), the QS
binary tracing protocol, **and most of the framework layer that the original analysis
flagged as missing**. Since the first revision, the hierarchical state machine framework
(`QHsm`), static event pools (`QMPool` + `q_new`/`gc`), the raw event queue and
defer/recall, ISR-safe posting, `QF::run()`/`stop()`, time-event `rearm()`/`wasDisarmed()`,
the QS semaphore/mutex records, the **QS-RX host→target command parser**, and **QUTest
test probes** have all landed. The Cortex-M port now performs real PendSV/SVC context
switching.

The remaining true gaps are **selective publish-subscribe** (publish still broadcasts to
all AOs), **multi-tick-rate** (single tick domain), and **true context switching for the
hosted QXK** (still a polling model; only the Cortex-M port switches contexts for real).

The Rust type system eliminates the need for several QP/C++ safety features (DIS integrity
checks, MISRA constraints) but requires deliberate API mapping to enable a smooth porting
experience.

## 1. State Machine Framework (QEP) — MAJOR GAP

### QP/C++ provides:
- **QHsm**: Code-based hierarchical state machine with automatic entry/exit/init sequencing across state hierarchy up to 6 levels deep
- **QMsm**: Table/model-driven HSM, generated by QM modeling tool, more efficient at runtime
- **QAsm**: Abstract base allowing polymorphic dispatch of `init()` / `dispatch()` / `isIn()`
- State handler return codes: `Q_RET_SUPER`, `Q_RET_HANDLED`, `Q_RET_TRAN`, `Q_RET_TRAN_HIST`, `Q_RET_IGNORED`
- History pseudo-states (`Q_RET_TRAN_HIST`)
- `childState()` / `childStateObj()` for introspection
- Macros: `Q_TRAN()`, `Q_SUPER()`, `Q_HANDLED()`, `Q_IGNORED()`
- Note: `QFsm` (flat SM) **removed** in v8.0.0 — migration target is `QHsm`

### qp-rs provides:
- `ActiveBehavior` trait with `on_start()` and `on_event()` — application defines its own state management
- `SignalHandler` convenience trait for simple flat state machines
- QS record type IDs defined for all state machine events (STATE_ENTRY, STATE_EXIT, TRAN, etc.) but no framework emits them automatically

### Gap:
No HSM dispatch algorithm, no automatic entry/exit chain execution, no hierarchy traversal, no `init()` pseudo-state sequencing. Applications must implement state machines by hand.

### Impact on drop-in replacement:
**Critical** — every QP/C++ application derives from `QHsm` or `QMsm`. Without an equivalent, porting requires complete rewrite of all state handlers.

## 2. Memory Management (QMPool / Event Pools) — MAJOR GAP

### QP/C++ provides:
- **QMPool**: Fixed-size block allocator with free-list, high-water mark tracking, DIS integrity
- **QF event pools**: Up to `QF_MAX_EPOOL` (default 3, max 15) pools of different block sizes
- `QF::poolInit(sto, size, blockSize)`: Register pool with framework
- `QF::q_new<T>(sig)` / `QF::q_new_x<T>(margin, sig)`: Typed event allocation from pool
- `QF::gc(e)`: Garbage collect (decrement refcount; free when zero)
- `QF::newRef_()` / `deleteRef_()`: Manual reference management
- ISR variants: `QF::newXfromISR_()`, `QF::gcFromISR()`
- `getPoolUse()`, `getPoolFree()`, `getPoolMin()`: Diagnostics
- **QEvtPtr** (v8.1.3+): Opaque wrapper for `QEvt const *` enabling safety instrumentation
- Queue storage typed as `QEvtPtr qSto[]` (breaking change from v8.1.3)

### qp-rs provides:
- `EventHeader` has `pool_id: Option<u8>` and `ref_count: u8` fields (infrastructure present)
- `Arc<DynPayload>` for zero-copy event sharing across AOs
- No explicit pool allocator; heap allocation via Rust's global allocator
- No `gc()` — Arc handles deallocation automatically

### Gap:
No static memory pool allocation. Cannot operate in `no_std` without `alloc`. No per-pool diagnostics. No equivalent to `QEvtPtr` wrapper. Arc reference counting semantics differ from QP's manual refcount (QP events can be on ISR stack; Arc requires heap).

### Impact on drop-in replacement:
**High** — embedded targets often have no heap. A pool-based allocator is required for bare-metal use. The `pool_id` field exists in EventHeader but is unused.

## 3. Active Object Lifecycle API — MODERATE GAP

### QP/C++ `QActive::start()` signature:
```cpp
void start(QPrioSpec prioSpec,  // Q_PRIO(prio, threshold) — dual encoding
           QEvtPtr *qSto,       // static queue storage array
           uint_fast16_t qLen,
           void *stkSto,        // stack (NULL for QK/QV, required for QXK)
           uint_fast16_t stkSize,
           void const *par = nullptr);  // init parameter passed to QHsm::init()
```

### QP/C++ additional lifecycle:
- `QActive::stop()` (optional, requires `QACTIVE_CAN_STOP`)
- `QF::psInit(subscrSto, maxSignal)`: Initialize pub-sub subscriber list
- `QF::onStartup()` / `QF::onCleanup()`: Application callbacks around `QF::run()`
- `QActive::register_()` / `unregister_()`: Internal AO registry
- `QActive::fromRegistry(prio)`: Lookup by priority

### qp-rs provides:
- Registration via kernel builder: `kernel.register(ao)` / `register_with_threshold(ao, threshold)`
- No `start()` on individual AOs — lifecycle managed by kernel
- No `stop()`
- `KernelConfig` provides equivalent to `qp_config.hpp` parameters
- No static queue storage (uses `VecDeque` heap allocation)
- No init parameter passed to state machine on construction

### Gap:
The `start()` API is the primary integration point for every QP application. The static queue storage pattern (essential for no_std) and the init parameter are both absent.

## 4. Publish-Subscribe — MODERATE GAP

### QP/C++ provides:
- `QActive::psInit(subscrSto, maxSignal)`: Static subscriber list allocation
- `QActive::subscribe(sig)` / `unsubscribe(sig)` / `unsubscribeAll()`
- `QActive::publish_(e, sender, qsId)`: Multicasts to all subscribers
- Subscription bitmaps; multicast protected by scheduler lock for atomicity
- `QS_QF_ACTIVE_SUBSCRIBE` / `QS_QF_ACTIVE_UNSUBSCRIBE` trace records

### qp-rs provides:
- `kernel.publish(signal, event)`: Broadcasts to all registered AOs unconditionally
- No subscription management — every AO receives every published event
- No subscribe/unsubscribe API

### Gap:
No selective subscription. All AOs receive all published events regardless of interest. On a system with many AOs and many signals, this wastes queue space and dispatch time.

## 5. Defer / Recall — MISSING

### QP/C++ provides:
```cpp
bool QActive::defer(QEQueue *eq, QEvt const *e);   // push to defer queue
bool QActive::recall(QEQueue *eq);                  // move front event to own queue
uint16_t QActive::flushDeferred(QEQueue *eq, uint_fast16_t num);
// QS records: QS_QF_ACTIVE_DEFER, QS_QF_ACTIVE_RECALL, 
//             QS_QF_ACTIVE_RECALL_ATTEMPT, QS_QF_ACTIVE_DEFER_ATTEMPT (v8.0.4)
```

### qp-rs provides:
- Nothing — no defer/recall mechanism

### Gap:
Common pattern in QP state machines for handling events that arrive before the SM is ready to process them (e.g., while in a transitional state).

## 6. Raw Event Queue (QEQueue) — MISSING

### QP/C++ provides:
```cpp
class QEQueue {
    void init(QEvtPtr *qSto, uint_fast16_t qLen);
    bool post(QEvt const *e, uint_fast16_t margin, uint_fast8_t qsId);
    void postLIFO(QEvt const *e, uint_fast8_t qsId);
    QEvt const *get(uint_fast8_t qsId);
    bool isEmpty(); QEvt const *peekFront();
    uint16_t getFree(); uint16_t getMin();
};
```

Used for: AO internal queues, deferred event queues, standalone raw queues (ISR→AO communication without active object wrapper).

### qp-rs provides:
- `VecDeque<DynEvent>` embedded in `ActiveObject` — not accessible externally
- No standalone raw event queue

## 7. Time Event API — MINOR GAP

### QP/C++ provides:
- `QTimeEvt(act, sig, tickRate)`: Constructor links event to AO + tick rate
- `armX(nTicks, interval)`: Arm with absolute count
- `disarm()` → bool (was armed?)
- `rearm(nTicks)` → bool: Change counter without disarming
- `wasDisarmed()`: Sticky flag, cleared on read
- `getCtr()` / `getInterval()`: Counters
- `noActive(tickRate)`: Static query — any armed TE for this tick rate?
- `tickFromISR(tickRate, par, sender)`: ISR-safe tick advance
- **Multi-tick-rate**: Up to 15 independent tick domains (default 1, max `QF_MAX_TICK_RATE`)
- **QTicker**: Convenience `QActive` subclass that forwards tick signals from ISR

### qp-rs provides:
- `TimeEvent::new(target_id, config)`: Creates with target AO and DynEvent
- `arm(timeout, interval)`: One-shot or periodic
- `disarm()`: Cancel
- `is_armed()`: State query
- `poll()`: Returns expired events for posting
- `TimerWheel` (QF) and `QkTimerWheel` (QK)
- **Single tick rate only** — no multi-tick-rate support

### Gap:
- `rearm()` (update counter without disarm/rearm cycle) missing
- `wasDisarmed()` sticky flag missing
- `noActive()` static query missing
- Multi-tick-rate (tick domains) not supported
- No `tickFromISR()` path
- No `QTicker` AO

## 9. QXK Extended Threads — MODERATE GAP

### QP/C++ QXThread provides:
- True stack-based context switching (assembly-level context save/restore)
- `QXThread::delay(nTicks)`: Block for N ticks (suspends the thread, not polling)
- `QXThread::queueGet(nTicks)`: Block until event arrives in thread's queue
- `QXSemaphore::wait(nTicks)` with real timeout (counter decremented on tick ISR)
- `QXMutex` with **priority ceiling protocol**: thread priority raised while mutex held
- ISR-safe `signal()` / `fromISR` variants

### qp-rs QXK provides:
- Polling-based cooperative threads — handler called repeatedly in dispatch loop
- `ThreadAction::Blocked` signals intent to block, but no real suspension
- No hardware-level context switch (no stack pointer manipulation, no ISP/PSP switching)
- `wait_timeout()` in Semaphore is stub-like
- `MutexPrim` has no priority inheritance or ceiling protocol
- No ISR-safe primitive variants

### Gap:
The qp-rs QXK is a cooperative thread emulation, not a true dual-mode preemptive+blocking kernel. On Cortex-M targets, QP/C++ QXK uses PendSV for context switches; qp-rs has no equivalent. For embedded porting, this is a fundamental architectural gap.

### Note on Rust tradeoffs:
True context switching in `no_std` Rust requires `unsafe` assembly for stack manipulation, which is achievable but non-trivial. The polling model works for many use cases but breaks QP/C++ applications that rely on `delay()` for time-based blocking.

## 10. QF Framework Lifecycle — MINOR GAP

### QP/C++ provides:
```cpp
QF::init();        // zero all framework data structures
QF::run();         // blocking: enter scheduler loop
QF::stop();        // signal framework to stop
QF::onStartup();   // user callback: called AFTER kernel init (added post v8.0.3)
QF::onCleanup();   // user callback: called before exit
```

### qp-rs provides:
- `Kernel::builder()` → `build()` → `start()`: Setup chain but no persistent `run()` loop
- `TimerWheel::tick()` / `dispatch_once()` / `run_until_idle()`: Manual run-loop composition
- No framework-wide `stop()` signal
- No `onStartup()` / `onCleanup()` callbacks

### Gap:
`QF::run()` is the main application entry point in all QP applications. qp-rs requires the user to compose the run loop manually.

## 11. Priority Encoding (QPrioSpec) — MINOR GAP

### QP/C++ v8.x:
```cpp
#define Q_PRIO(prio_, pthre_)  ((prio_) | ((pthre_) << 8U))
// Single uint16_t encodes both base priority and preemption threshold
// Passed to QActive::start()
```

### qp-rs:
- Separate `register_with_threshold(ao, threshold)` on kernel builder
- No `QPrioSpec` type or `Q_PRIO` macro equivalent
- Priority and threshold specified separately at registration time

## 12. ISR Integration — GAP

### QP/C++ provides ISR-safe variants for all operations:
- `QActive::postFromISR()`, `publishFromISR()`
- `QTimeEvt::tickFromISR()`
- `QF::newXfromISR_()`, `gcFromISR()`
- `QMPool::getFromISR()`, `putFromISR()`
- `QK_ISR_ENTRY()` / `QK_ISR_EXIT()` macros for ISR nesting tracking

### qp-rs:
- No ISR-specific API variants
- No interrupt nesting counter
- `ports/esp32-*` interrupt controllers are stubs
- No `QK_ISR_ENTRY`/`EXIT` equivalent

### Impact: Required for any real embedded target.

## 13. Diagnostics / Watermarks — MINOR GAP

### QP/C++ provides per-resource diagnostics:
- `QActive::getQueueUse/Free/Min(prio)`: Per-AO queue watermarks
- `QF::getPoolUse/Free/Min(poolNum)`: Per-pool watermarks
- `QEQueue::getMin()`: Per-queue watermarks
- `QXSemaphore` / `QXMutex` usage tracking

### qp-rs:
- No queue watermark tracking
- No pool diagnostics (no pools)
- No resource usage APIs

## 14. QUTest — NOT IMPLEMENTED

QP/C++ ships an embedded unit testing framework built into the QS infrastructure:
- Test setup/teardown via QS-RX commands
- `QS_TEST_PROBE()` injection points in production code
- Python-based test scripts via QSpy
- On-target and host-based execution

qp-rs has no equivalent. Testing relies on standard Rust test harness.

## 15. Features NOT Needed in Rust

These QP/C++ features have Rust equivalents or are unnecessary:

| QP/C++ Feature | Rust Equivalent | Notes |
|---|---|---|
| DIS (Duplicate Inverse Storage) integrity | Rust type system + ownership | Memory safety by construction |
| MISRA-C++:2023 compliance | Clippy lints + safe Rust | Different mechanism, similar goals |
| `volatile` on safety-critical fields | `core::sync::atomic` | More explicit in Rust |
| `Q_ASSERT()` / `Q_ERROR()` | `assert!()` / `panic!()` | Rust panics; no manual handler needed |
| QFsm (removed in v8.0.0) | Not applicable | Already removed upstream |
| Manual `QF_bzero()` zero-initialization | Rust default initialization | Types always initialized |
| `qp_config.hpp` mandatory config file | `KernelConfig` builder | Already implemented |
| `QEvtPtr` struct wrapper | Rust newtype pattern | If implementing pools |

## Phased Implementation Plan

### Phase 1 — API Surface & Lifecycle (Foundation) [~2 weeks]

Goals: Make the framework entry points match QP/C++ conventions.

1. **`QF::run()` loop abstraction** (`crates/qf/src/kernel.rs`)
   - Add `Kernel::run(tick_fn: impl FnMut())` as a blocking run-loop primitive
   - Add `Kernel::stop()` with stop flag
   - Add `on_startup` / `on_cleanup` callbacks to `KernelConfig`

2. **`QPrioSpec` type** (`crates/qf/src/active.rs`, `crates/qk/src/kernel.rs`)
   - Add `QPrioSpec(u16)` newtype with `Q_PRIO(prio, threshold) -> QPrioSpec` function
   - Accept `QPrioSpec` in kernel registration

3. **`QTimeEvt::rearm()` + `wasDisarmed()`** (`crates/qf/src/time.rs`)
   - Add `rearm(n_ticks)` to update counter without disarm cycle
   - Add `was_disarmed()` sticky flag
   - Add `no_active(tick_rate) -> bool` static query

4. **Publish-subscribe** (`crates/qf/src/kernel.rs`)
   - Add `ps_init(max_signal)` on kernel
   - Add `subscribe(signal)` / `unsubscribe(signal)` on active objects
   - Filter `publish()` through subscription bitmaps

5. **Multi-tick-rate** (`crates/qf/src/time.rs`, `crates/qk/src/time.rs`)
   - Parameterize `TimerWheel` with tick_rate index (0..N)
   - Allow multiple `TimerWheel` instances; `KernelConfig::max_tick_rate` enforced

### Phase 2 — Hierarchical State Machine (QHsm) [~3–4 weeks]

Goals: Provide a QHsm-compatible framework so state machine code can port with minimal changes.

1. **`QHsm` trait / struct** (new file: `crates/qf/src/hsm.rs`)
   - Define `StateHandler` as `fn(&mut Self, &DynEvent) -> QHsmResult`
   - `QHsmResult` enum: `Super(StateHandler)`, `Handled`, `Tran(StateHandler)`, `TranHist(StateHandler)`, `Ignored`, `Entry`, `Exit`, `Init`
   - `QHsm<S>` struct: holds current state handler + initial state
   - `init(event)`: drive top-most initial transition, enter hierarchy to leaf
   - `dispatch(event)`: walk up hierarchy via `Q_RET_SUPER` until handled; execute exit→entry

2. **Hierarchy traversal algorithm**
   - `MAX_NEST_DEPTH = 6` limit (compile-time)
   - `tran_simple_()`: transition within same composite state (shared ancestor)
   - `tran_complex_()`: find LCA, exit chain to LCA, enter chain to target
   - `childState(parent)` introspection

3. **History pseudo-states**
   - Shallow history: store last active substate per composite state

4. **Automatic QS record emission**
   - `STATE_ENTRY`, `STATE_EXIT`, `STATE_INIT`, `TRAN`, `INTERN_TRAN`, `IGNORED`, `DISPATCH`, `UNHANDLED`, `TRAN_HIST`

5. **`ActiveBehavior` bridge**
   - Provide blanket impl of `ActiveBehavior` for any `QHsm<S>`
   - Existing `ActiveBehavior` trait remains for custom state machine implementations

6. **Macros** (optional ergonomics)
   - `q_tran!(state)`, `q_super!(state)`, `q_handled!()`, `q_ignored!()`

### Phase 3 — Memory Management (Event Pools) [~2–3 weeks]

Goals: Enable static memory operation for embedded targets.

1. **`QMPool`** (new file: `crates/qf/src/pool.rs`)
   - Fixed-size block allocator over a `&'static mut [u8]` slice
   - Free list via embedded next pointers
   - `init(storage, pool_size, block_size)` 
   - `get(margin) -> Option<*mut u8>` / `put(block)`
   - Watermark tracking: `get_min()`, `get_use()`, `get_free()`
   - `unsafe` internals; safe public API via block ownership tracking

2. **Framework event pools**
   - `KernelConfig::pool_init(storage, block_size)` registers up to N pools
   - `q_new<T>(sig) -> EventBox<T>`: Allocates from smallest fitting pool
   - `q_new_x<T>(margin, sig) -> Option<EventBox<T>>`: Returns None if margin not met
   - `EventBox<T>`: Smart pointer that calls `gc()` on drop
   - `gc(event)`: Decrement pool refcount; free when zero

3. **`no_std` allocation strategy**
   - With `alloc` feature: fall back to `Box` allocation
   - Without `alloc`: pool allocator is mandatory

4. **QS records**: MPOOL_INIT, MPOOL_GET, MPOOL_PUT, NEW, GC, NEW_ATTEMPT, GC_ATTEMPT, MPOOL_GET_ATTEMPT

### Phase 4 — Defer/Recall + QEQueue [~1 week]

Goals: Complete event queuing infrastructure.

## Architectural Notes for Rust Port

### Where Rust improves on QP/C++:
1. **Memory safety eliminates DIS checks** — No need for bitwise-inverse integrity fields; the borrow checker prevents aliasing
2. **Arc replaces manual refcount** — `QEvt::poolNum_` + `refCtr_` pattern replaced by `Arc<>` (can remove when pools added)
3. **Type safety for events** — `Event<T>` + `downcast_ref` is cleaner than C++ casting
4. **No `qp_config.hpp`** — `KernelConfig` builder with type-safe defaults
5. **`no_std` via features** — More explicit than C++ `#ifdef`

### Where Rust complicates QP/C++ patterns:
1. **State handlers as function pointers** — C++ `QStateHandler` is a plain function pointer; Rust state machines need careful handling of `&mut self` captures
2. **Static event queue storage** — C++ `QEvtPtr qSto[N]` is a simple stack/BSS array; Rust needs `&'static mut` or `MaybeUninit` wrappers
3. **ISR-safe code** — C++ uses `QF_INT_DISABLE()`; Rust needs `core::sync::atomic` and architecture-specific critical section primitives
4. **True context switching** — C++ QXK uses `setjmp`/`longjmp` or assembly; Rust needs `unsafe` assembly blocks
5. **Mutable global state** — C++ uses global objects freely; Rust needs `static Mutex<>` or `lazy_static` patterns
